TL;DR — kōdo is a universal memory layer for AI coding agents. It stores typed memories (conventions, mistakes, decisions, patterns) in a local SQLite database, shares them across Claude Code / Cursor / Kiro / Codex via MCP, and broadcasts discoveries between parallel terminal sessions in real time via a pub/sub hub. Self-evolution prunes dead memories, merges duplicates, and promotes high-value knowledge automatically.

The Amnesia Problem

Every AI coding agent starts every session with a blank slate. You correct its code style — it forgets next session. You fix a bug — same bug tomorrow. You switch from Cursor to Claude Code — start over from scratch.

System Architecture

🏗️
Figure 2. kōdo architecture diagram — memory store, MCP server, cross-terminal hub, agent exports.
(auto-generated by research pipeline)

kōdo consists of four components:

  1. Memory Store — SQLite + FTS5 full-text search. Typed memories: convention, mistake, decision, preference, pattern, note.
  2. MCP Server — Agents read/write memories live during sessions via Model Context Protocol.
  3. Cross-Terminal Hub — Unix domain socket pub/sub. When terminal A learns something, terminal B knows instantly.
  4. Agent Export — One command generates native config files for Claude Code, Cursor, Kiro, and Codex.

Self-Evolving Memory

🧬
Figure 4. Memory quality over time — evolution cycles showing pruning, merging, and promotion effects.
(auto-generated by research pipeline)

Inspired by CORAL's autonomous multi-agent evolution, kōdo's memory store self-evolves:

  • Prune — Remove memories with zero access after 30 days
  • Merge — Consolidate near-duplicate memories (Jaccard similarity > 0.7)
  • Promote — Boost relevance score of frequently accessed memories
